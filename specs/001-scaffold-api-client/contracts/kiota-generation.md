# Contract: Kiota Generation

Defines the inputs, outputs, and invariants of running the Kiota generator
against `openapi.json`. The actual invocation is wrapped by
`scripts/regenerate-buildin-client.sh` (see `regeneration-script.md`).

## Inputs

| Input | Source | Notes |
|---|---|---|
| OpenAPI document | `openapi.json` at repo root | OpenAPI 3.1.0; not modified by generation. |
| Kiota CLI version | pinned in `.config/dotnet-tools.json` | All contributors run identical generator. |
| Generator flags | hard-coded in the regeneration script | Must be deterministic across machines. |

## Generator invocation (canonical form)

```bash
dotnet kiota generate \
  --language CSharp \
  --openapi ./openapi.json \
  --class-name BuildinApiClient \
  --namespace-name Buildout.Core.Buildin.Generated \
  --output ./src/Buildout.Core/Buildin/Generated \
  --clean-output \
  --exclude-backward-compatible \
  --log-level Warning
```

Flag rationale:

- `--clean-output`: removes stale files; keeps the directory consistent with
  the current schema.
- `--exclude-backward-compatible`: smaller, cleaner generated code; we control
  Kiota's version via `dotnet-tools.json`, so we do not need backward-compat
  shims.
- `--log-level Warning`: keeps regeneration logs low-noise; warnings still
  surface (e.g. unsupported `oneOf` shapes).

## Outputs

| Output | Path | Tracked in git? |
|---|---|---|
| Generated `.cs` files | `src/Buildout.Core/Buildin/Generated/**/*.cs` | Yes |
| `kiota-lock.json` | `src/Buildout.Core/Buildin/Generated/kiota-lock.json` | Yes |
| Generator marker README | `src/Buildout.Core/Buildin/Generated/_README.md` | Yes (hand-written, NOT regenerated) |

## Invariants

1. **Determinism**: Two consecutive runs against an unchanged `openapi.json`
   produce zero diff in `git status`. (Spec FR-006, SC-004.)
2. **Containment**: All file changes from a regeneration run land inside
   `src/Buildout.Core/Buildin/Generated/`. No file outside this directory may
   change as a result of regeneration. (Spec FR-007, SC-005.) The
   `_README.md` is the lone exception — it is hand-written and only updated
   by manual edit; regeneration MUST NOT overwrite it.
3. **Auto-generated header**: every emitted `.cs` file has a Kiota-emitted
   `<auto-generated/>` header (Kiota default). Analyzers and IDE refactorings
   honour this header.
4. **Internal-only namespace**: the generated namespace
   `Buildout.Core.Buildin.Generated` is treated as internal. Public consumers
   reach buildin functionality only through `IBuildinClient`. This is
   enforced by:
   - Kiota's `--internal-classes` configuration where supported, OR
   - an `InternalsVisibleTo` declaration limiting visibility to
     `Buildout.Core` itself plus the test projects.
5. **Build hygiene**: generated files compile cleanly under the solution-wide
   nullable + warnings-as-errors policy. Where Kiota output triggers spurious
   warnings, `Directory.Build.targets` excludes the `Generated/` subtree
   from analyzers and code-style enforcement (R14).

## Failure modes

| Failure | Detection | Response |
|---|---|---|
| `openapi.json` has a malformed schema Kiota cannot parse | Kiota CLI exits non-zero | Surface the error to the user; regeneration script aborts; no partial overwrite (relies on `--clean-output` + git's working-tree integrity). |
| Kiota produces a non-determinism diff | `git status` after regen is dirty when input is unchanged | Investigate; usually a Kiota version drift (the local-tool pin should prevent this). |
| Generated code triggers warnings as errors | `dotnet build` fails | `Directory.Build.targets` excludes `Generated/` from style checks; if it's still failing, file an upstream Kiota issue. |
| Operation appears in `openapi.json` but not in generated client | Inspection / unit test for OperationId presence | Investigate `--exclude-path` / `--include-path` filters; we currently use neither, so a missing operation is a Kiota bug or an OpenAPI document defect. |
