# Implementation Plan: Scaffold + Buildin API Client

**Branch**: `001-scaffold-api-client` | **Date**: 2026-05-04 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/001-scaffold-api-client/spec.md`

## Summary

Bootstrap the buildout repository with the constitution-mandated five-project .NET 10
layout (`Buildout.Core`, `Buildout.Mcp`, `Buildout.Cli`, `Buildout.UnitTests`,
`Buildout.IntegrationTests`) plus a hand-written `IBuildinClient` abstraction in
`Buildout.Core`. The Bot-API implementation of that interface wraps a typed client
generated by **Kiota** from the repo's `openapi.json`. Generated code is committed
under a marked subdirectory and regenerable via a single script. All tests use
in-memory mocks (`IRequestAdapter` for unit tests, a custom `HttpMessageHandler` for
integration tests) — no test makes a real network call to buildin.ai. Presentation
projects (`Buildout.Mcp`, `Buildout.Cli`) ship as buildable shells; no MCP tools or
CLI commands land in this feature.

## Technical Context

**Language/Version**: C# / .NET 10 (per constitution Technology Standards)
**Primary Dependencies**:

- `Microsoft.Kiota.Bundle` — generated-client runtime (HTTP, serialization, auth)
- `Microsoft.OpenApi.Kiota` — CLI generator, pinned via local `dotnet-tools.json`
- `Microsoft.Extensions.Configuration` (+ `.EnvironmentVariables`, `.Binder`) — config sources
- `Microsoft.Extensions.Logging` — logging abstractions
- `Microsoft.Extensions.DependencyInjection` — DI for client + options binding
- `ModelContextProtocol` — referenced by `Buildout.Mcp` shell; no transports wired in this feature
- `Spectre.Console.Cli` — referenced by `Buildout.Cli` shell; no commands wired in this feature
- `xUnit.v3`, `NSubstitute` — test framework + mocking lib (matches official Kiota testing guide)

**Storage**: N/A — no persistence in this feature.
**Testing**: xUnit; unit tests mock `IRequestAdapter` via NSubstitute; integration tests inject a custom `HttpMessageHandler` into Kiota's `HttpClient`. Both run fully in-process.
**Target Platform**: cross-platform .NET 10 (macOS, Linux, Windows).
**Project Type**: single .NET solution producing a library (`Buildout.Core`) and two console-style apps (`Buildout.Mcp`, `Buildout.Cli`), plus two test projects.
**Performance Goals** (from spec SCs):

- Clone → green build + green tests in ≤ 5 min on a fresh machine with .NET SDK installed (SC-001).
- Full test suite completes in < 60 s on a developer laptop, no network (SC-003, SC-006).

**Constraints**:

- No outbound HTTPS to `api.buildin.ai` from any test (Constitution Principle IV; spec FR-008, SC-006).
- Nullable reference types + warnings-as-errors solution-wide (Constitution Technology Standards).
- Generated code lives only inside `src/Buildout.Core/Buildin/Generated/` (spec FR-005, FR-007).
- Regeneration is deterministic — two runs against unchanged `openapi.json` produce zero diff (spec FR-006, SC-004).
- No secrets in source; bot tokens injected at construction via configuration (spec FR-010; constitution Secrets).

**Scale/Scope**: 15 operations across 10 paths × ~80 schemas in `openapi.json`; presentation projects ship as buildable shells (no commands/tools).

**Structure Decision**: Single .NET 10 solution (`buildout.slnx`) with the
constitution-mandated five projects. Kiota-generated client output is committed under
`src/Buildout.Core/Buildin/Generated/` and regenerated via
`scripts/regenerate-buildin-client.sh`. The generated client is wrapped by
`BotBuildinClient`, which is the only place that knows about Kiota's wrapper-class
treatment of `oneOf`/`anyOf` schemas (see `research.md` R12). Downstream features
depend only on `IBuildinClient` and the hand-written domain types in
`Buildout.Core/Buildin/Models/`.
