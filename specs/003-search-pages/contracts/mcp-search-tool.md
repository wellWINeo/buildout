# Contract — MCP `search` tool

Located at `src/Buildout.Mcp/Tools/SearchToolHandler.cs`. Registered in
`Buildout.Mcp/Program.cs` via `WithTools<SearchToolHandler>()`.

## Tool descriptor

| Field | Value |
|---|---|
| Name | `search` |
| Title | `Search buildin pages by query` |
| Description | `"Search buildin pages by query. Returns one match per line, tab-separated: <page_id>\\t<object_type>\\t<title>. Use buildin://<page_id> to read a match."` |
| Input schema | See below |
| Output | Single text content block; body = `ISearchResultFormatter.Format(matches)` |

### Input schema

```json
{
  "type": "object",
  "properties": {
    "query":   { "type": "string", "description": "Non-empty search query." },
    "page_id": { "type": "string", "description": "Optional buildin page UUID. When set, restricts results to descendants of this page." }
  },
  "required": ["query"]
}
```

The schema is generated by the `ModelContextProtocol` SDK from the
handler method's parameter signature and `[Description]` attributes:

```text
[McpServerToolType]
public sealed class SearchToolHandler
{
    [McpServerTool(Name = "search")]
    [Description("Search buildin pages by query. Returns one match per line, tab-separated: <page_id>\\t<object_type>\\t<title>. Use buildin://<page_id> to read a match.")]
    public async Task<string> SearchAsync(
        [Description("Non-empty search query.")] string query,
        [Description("Optional buildin page UUID. When set, restricts results to descendants of this page.")] string? page_id = null,
        CancellationToken cancellationToken = default)
    { … }
}
```

The SDK serialises the return `string` into a single
`TextContentBlock(Text = <returned-string>)` inside the resulting
`CallToolResponse`.

## Behavioural contract

### Inputs

- `query` — required, non-empty. The handler validates and rejects
  empty / whitespace queries before calling the service. (The service
  also validates; the handler's check is for a clearer MCP-level
  error.)
- `page_id` — optional. When omitted or null, search is unscoped.
  When provided, must be a buildin page id `IBuildinClient.GetPageAsync`
  accepts. Format validation is delegated to the client.
- `cancellationToken` — propagated by the SDK from the MCP request
  cancellation.

### Outputs

A `string` whose value is exactly the body of
`ISearchResultFormatter.Format(matches)`:

- **Successful search with matches**: one line per match, three
  tab-separated columns ending in `\n`. See
  `contracts/search-result-format.md` for the full grammar.
- **Successful search with zero matches**: empty string `""`.
- This response is byte-identical to the CLI's plain-mode stdout for
  the same query and same buildin-response fixture (FR-014, SC-003).

### Error contract (FR-015)

Failures are surfaced as `McpProtocolException` thrown from the handler:

| Cause | Thrown |
|---|---|
| Empty / whitespace `query` | `McpProtocolException("Query must be non-empty.", McpErrorCode.InvalidParams)` |
| `BuildinApiException` with `ApiError { StatusCode: 404 }` (ancestor walk for `page_id`) | `McpProtocolException($"Page not found: {page_id}", McpErrorCode.ResourceNotFound)` |
| `BuildinApiException` with `ApiError { StatusCode: 401 or 403 }` | `McpProtocolException($"Authentication error: {ex.Message}", McpErrorCode.InternalError)` |
| `BuildinApiException` with `Error is TransportError` | `McpProtocolException($"Transport error: {ex.Message}", McpErrorCode.InternalError)` |
| Any other `BuildinApiException` | `McpProtocolException($"Unexpected buildin error: {ex.Message}", McpErrorCode.InternalError)` |
| `OperationCanceledException` | Re-thrown unchanged (the SDK handles cancellation) |

The handler MUST NOT return a successful `CallToolResponse` whose body
contains an error description; FR-015 forbids that. Any caller-visible
problem is an MCP-protocol error.

### Determinism

For a fixed input + fixed buildin-response fixture, the tool returns a
byte-identical body on every invocation. Verified by a test that calls
the tool twice via the in-process MCP client and compares results.

## Registration

`Buildout.Mcp/Program.cs`:

```text
builder.Services.AddMcpServer()
    .WithStdioServerTransport()
    .WithResources<PageResourceHandler>()    // existing (feature 002)
    .WithTools<SearchToolHandler>();         // NEW (this feature)
```

The handler is constructed by DI with `ISearchService`,
`ISearchResultFormatter`, and `ILogger<SearchToolHandler>` as
dependencies — all already registered by `AddBuildoutCore`.

## Test obligations

Integration tests live at
`tests/Buildout.IntegrationTests/Mcp/SearchToolTests.cs`. The harness
mirrors `PageResourceTests` from feature 002: in-process MCP server +
client, `IPipelinePair` streams, `IBuildinClient` substituted via
`NSubstitute`.

| Test | Asserts |
|---|---|
| `ServerAdvertisesSearchTool` | `ListToolsAsync` returns one tool whose Name is `search`, with the documented description and schema |
| `Search_ReturnsFormattedBody` | Calling the tool with a fixture-backed query returns a single text content block whose body equals the formatter's output |
| `Search_NoMatches_ReturnsEmptyBody` | Tool result content body is `""` |
| `EmptyQuery_ThrowsMcpInvalidParams` | Empty query → `McpProtocolException` with `InvalidParams` code; substituted client recorded zero `SearchPagesAsync` calls |
| `Scope_NotFound_ThrowsResourceNotFound` | `page_id = "missing"` where ancestor `GetPageAsync` returns 404 → `McpProtocolException` with `ResourceNotFound` |
| `AuthFailure_ThrowsInternalError` | 401 / 403 from `SearchPagesAsync` → `InternalError` |
| `TransportFailure_ThrowsInternalError` | `TransportError` → `InternalError` |
| `ToolResultBody_EqualsCliPlainBody` | Run CLI plain mode and MCP tool against the same fixture; assert byte-identical bodies (the SC-003 invariant) |
| `MultiplePages_AreDrained` | Substituted `SearchPagesAsync` returns three pages each with `HasMore = true` chained by `NextCursor`; final tool result contains all matches |
| `Determinism_TwoInvocations_ReturnIdenticalBody` | Two consecutive calls with the same fixture return byte-identical bodies |

The cheap-LLM test that exercises the tool through Claude Haiku lives
in `tests/Buildout.IntegrationTests/Llm/PageReadingLlmTests.cs` as
`LlmCanFindAndReadPage` (see `research.md` R10). It is a separate test
class (existing from feature 002) and is gated on `ANTHROPIC_API_KEY`.

## Out of scope (deferred to later features)

- A `read` tool variant of `buildin://{page_id}` resource. Reading
  remains a resource (per feature 002); search is the only addition
  here on the tool surface.
- Returning structured (`matches: [...]`) JSON content. Plain text
  body matches the CLI by construction; structured shapes can be
  added additively later if real demand emerges.
- Filter / sort knobs (`include_archived`, `object_type`, etc.).
  Deferred per spec.
